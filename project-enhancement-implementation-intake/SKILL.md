# Project Enhancement Implementation Intake

この skill は、実装を始める直前に、今回守るべき実装契約を固定するために使う。  
目的は、仕様不足や対象外作業の混入を防ぎ、現在のワークスペースへ安全に直接反映できる状態にすることである。

## 入力
- `030-<Feature>実装指示書.md`
- `020-<Feature>実装計画書.md`
- 必要時のみ `010-<Feature>実装仕様書.md`
- 必要時のみ `005-<Feature>現行仕様整理.md`
- 必要時のみ `000-<Feature>前提メモ.md`
- 必要時のみ `project-code-grounding-analysis` の結果
- 必要時のみ reference routing / intake 結果

## 手順
1. `030` と `020` が揃っていることを確認する。
2. `030` と `020` から、対象機能、変更範囲、対象外、実装順序、対象ファイル、テスト観点、禁止事項を抽出する。
3. 不足または矛盾がある場合のみ、`010`、`005`、`000` を追加参照する。
4. 現在のワークスペース内で関連ファイルを読み、指示と実コードが矛盾していないか確認する。
5. 対象ファイル、根拠、既存変更単位が不明な場合は、実装前確認として `project-code-grounding-analysis` に戻す。
6. 既存構成にない DTO / Repository / Service / DI 分離などが指示されている場合は、その理由が `030 / 010` に明記されているか確認する。
7. 実装前に実装契約を短く作成する。
8. 実装契約が固まるまでコード変更を始めない。

## 停止条件
次の場合はコード変更を始めず、要確認事項として止める。

- `030` または `020` が見つからない。
- 対象機能、変更範囲、対象外、実装順序のいずれかが判断できない。
- 仕様資料と実コードの挙動が矛盾している。
- 対象ファイルが特定できず、変更影響を説明できない。
- 既存にない新規抽象化が指示されているが、その理由が仕様資料にない。
- 機密情報、接続文字列、固定宛先などをコードへ追加する必要があるように見える。

## 完了条件
- 何を変えてよく、何を変えてはいけないかが明確である。
- 既存コードと仕様資料に矛盾がない、または矛盾が要確認事項として分離されている。
- 実装順序とテスト観点が明確である。
- 別プロジェクト作成や対象外リファクタを行わないことが確認されている。
- 既存構成にない抽象化を追加する場合は、仕様上の根拠が明確である。
