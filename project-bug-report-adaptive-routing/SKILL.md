# Project Bug Report Adaptive Routing

初回調査でどこまで読んでよいかを決めるルーティング専用スキル。

## 目的

- 症状に直結する入口候補だけを見る
- プロジェクト誤判定と無駄な広域探索を抑える
- 後続スキルに渡す調査境界を固定する
- `project-bug-report-adaptive-token-guard` の上限内で調査する

## 判定材料

- チケット文面の固有語
- solution 名、project 名、浅い階層のディレクトリ名
- エラーメッセージ、スタックトレースの namespace、class、method
- 画面名、API名、ジョブ名、テスト名
- 既知の project reference 名

## 手順

1. 入力から対象ドメイン語と入口候補語を抽出する
2. ワークスペースの浅い情報だけで project 候補を最大2件まで絞る
3. 対応する reference を必要最小限だけ読む
4. 画面 / API / ジョブ / テストの入口候補を最大5ファイル程度に絞る
5. 入口から1段階先の呼び出し先まで見て、再現経路候補を簡潔に整理する

## 出力

1. 対象プロジェクト候補
2. 確信度
3. 読んだ reference
4. 入口ファイル候補
5. 再現経路の仮説
6. 次に見るべき最大3ファイル
7. ここで探索を止める条件

## 探索制約

- 初回から全体検索へ広げない
- 初回で読む reference は最大2件を基本とする
- 初回に読むファイルは最大5件を基本とする
- 1回に読む本文は原則100行以内に抑える
- `.git`、`node_modules`、`bin`、`obj`、生成物、大容量ログは除外する
- 対象が絞れない場合は、単一前提で断定しない
- 入口候補の役割説明は短く保ち、詳細原因分析へ踏み込まない
- 大容量ファイルは全文読みせず、検索ヒット周辺だけ確認する
