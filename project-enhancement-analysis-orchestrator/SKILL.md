# Project Enhancement Analysis Orchestrator

既存コードを調査し、実装に必要な仕様書・計画書・実装指示書を作成する。  
トークンおよび AI クレジットの過剰消費を避けるため、全探索を禁止し、サンプリングとフェーズ分割を優先する。

## ゴール
対象範囲が明確で、ユーザーが分析成果物の作成まで求めている場合は、以下を揃える。

1. 対象機能が現在何をしているかの整理
2. 要件と現行仕様の差分整理
3. 対象プロジェクトの既存構成に合わせた実装仕様
4. 実装と検証引き継ぎまでを対象にした実装計画
5. 実装 skill 向けの実装指示
6. 継続利用できる spec / progress 構成

対象範囲が曖昧、影響範囲が広い、入口候補が複数ある、またはユーザー指示が「分析・仕様・実装」を一括で含む場合は、まずフェーズ 1 としてサンプリング分析だけを行い、実装へ進まず停止する。対象範囲が明確で小さい場合は、同一フェーズ内で `010 / 020 / 030` などの分析成果物を作成してよい。

成果物は原則として `.workstate/<Project>/<Feature>/spec/` に作成する。

- `000-<Feature>前提メモ.md`
- `005-<Feature>現行仕様整理.md`
- `010-<Feature>実装仕様書.md`
- `020-<Feature>実装計画書.md`
- `030-<Feature>実装指示書.md`

進捗情報は `.workstate/<Project>/<Feature>/progress/` に作成または更新する。

- `001-詳細タスク.md`
- `002-進捗サマリ.md`
- `003-意思決定ログ.md`
- `004-保留事項一覧.md`
- `005-検証メモ.md`
- `006-次回着手メモ.md`
- `007-agent-skill-improvement-log.md`

## 最重要前提
- 対象 root は原則として現在のワークスペース root とする。
- ユーザーが別 root を明示した場合のみ、その root を対象にする。
- 対象プロジェクト自身を直接調査する。
- 別ディレクトリに一時的な開発用プロジェクトを作らない。
- 移行元プロジェクト、旧ソース root、作り直し前提を要求しない。
- 既存コードを読まずに仕様を決めない。
- 改修目的に不要な大規模リファクタ、最適化、技術刷新を混ぜない。
- 調査のためにビルド、テスト、アプリ起動を行わない。必要な場合は人間へ確認として渡す。

## コスト防衛ガードレール
このセクションは、他の分析手順より優先する。

### 1. 行動前の最小化判断
ファイル読み込み、検索、コマンド、別 skill 起動、成果物更新の直前に、次を満たす方法を選ぶ。

- その行動が、現行仕様の確認または実装指示作成に本当に必要か。
- ファイル全体ではなく、grep、該当行、head / tail、差分、設定名、ルート名だけで判断できないか。
- この 1 回で、次のフェーズに必要な情報をコンパクトに回収できるか。

「念のため全部読む」「まず全体を探索する」は禁止する。

### 2. 全探索禁止とサンプリング
- リポジトリ全体の全ファイル精査、自律的な Explore、全索子を禁止する。
- 初回調査は、README、`.sln`、`.csproj`、設定、入口候補、代表的な関連ファイルのうち、最大 3〜5 ファイル程度を目安にする。
- 対象機能、入口、成果物 root が明確な場合は、必要な範囲で 3〜5 ファイルを超えてもよい。
- 対象が未確定のまま広がる場合は、それ以上読まずに追加調査候補として止める。
- 代表ファイルのサンプリングで傾向を整理し、対象が広がる場合はそこで停止してユーザー確認へ回す。
- 入口候補が複数残る場合は、深掘り前に候補と判断理由を提示する。
- `node_modules`、`.git`、`bin`、`obj`、`dist`、`packages`、生成物、大容量ログは読まない。

### 3. 読み込み・コマンド出力の制限
- 1 回の表示・読み込み・コマンド出力で 100 行を超える情報を取得しない。
- `rg`、`Select-String`、`Get-Content | Select-Object -First`、`git diff --name-only`、`git status --short` など、低出力の確認を優先する。
- コマンド実行回数を増やさず、検索語と対象ディレクトリを絞って実行する。
- 全文 diff、全ログ、全設定出力、全テスト一覧出力は避ける。
- ビルド、テスト、アプリ起動、DB 接続、外部 API 呼び出しは行わない。

### 4. バックグラウンドエージェント禁止
- 大規模分析が必要でも、バックグラウンドエージェントや Explore agent を自動起動しない。
- 起動が必要な場合は、想定トークンまたはクレジット消費、目的、対象範囲を示し、ユーザーの明示許可を得る。
- 許可がない場合は、サンプリング分析結果と追加調査案だけを残して停止する。

### 5. フェーズ分割
- 「分析」「仕様検討」「実装指示作成」「コード生成 / 実装」を一括で進めない。
- 対象範囲が広い場合は、フェーズ 1 として現行把握・入口候補・追加調査範囲だけを出して停止する。
- ユーザーが次工程を明示した場合のみ、仕様書・計画書・指示書作成へ進む。
- すでに対象機能、入口、要件、成果物 root が明確で小さい場合は、同一フェーズ内で spec / progress を作成してよい。

### 6. progress 更新の集約
- progress ファイルを調査途中に細かく更新しない。
- `001-詳細タスク.md`、`002-進捗サマリ.md`、`003-意思決定ログ.md`、`004-保留事項一覧.md`、`005-検証メモ.md`、`006-次回着手メモ.md`、`007-agent-skill-improvement-log.md` の更新は、フェーズ末尾に 1 回だけまとめる。
- サンプリング分析だけで停止する場合は、無理に全成果物を作らず、次に読むべき最小ファイルと未確定事項を残す。

## 利用する skill
以下の skill を順に使う。ただし、対象が曖昧または広範囲の場合は、フェーズ 1 のサンプリング分析で停止する。

1. `project-reference-router`
   - 現在のワークスペース、要件、`.sln`、`csproj`、ディレクトリ名から読むべき reference を選ぶ。
   - reference がない場合は `reference_status: not_found` または `unavailable` として扱い、実コード調査へフォールバックする。
   - reference 探索は必要最小限に留める。
2. `project-reference-intake`
   - 選ばれた reference を読み、プロジェクト構成・既存流儀・注意点を仮説として整理する。
   - reference がない場合はスキップしてよい。
   - reference は仮説であり、仕様判断の正本にしない。
3. `project-code-grounding-analysis`
   - 実コードを正として、reference 仮説の裏取り、入口、呼び出し経路、入出力、副作用、設定、テストを整理する。
   - 全探索ではなく、対象機能に関係する代表箇所だけを確認する。
4. `project-enhancement-analysis`
   - 実コードで確認した事実をもとに、現行仕様、影響範囲、未確定事項を整理する。
5. `project-enhancement-implementation-spec`
   - 実装仕様書、実装計画書、実装指示書を作成する。
   - 実装時に追加判断が必要にならない粒度を目指すが、推測で埋めない。
6. `project-enhancement-progress-pack`
   - spec / progress の置き場、再開情報、判断ログ、agent / skill 改善ログをフェーズ末尾に 1 回だけ整理する。

既存の移行専用 skill は使わない。名前や前提が似ていても、移行元 root、旧ソース、Functions 移行を前提にした skill へ委譲してはならない。

## 判断原則
- Reference-first, Code-authoritative.
- Reference は調査の地図であり、仕様判断の正本ではない。
- 実コードで裏取りできない内容を確定仕様にしてはならない。
- Reference と実コードが矛盾する場合は、実コードを優先し、差分を記録する。
- 最終判断の優先順位は `実コード > reference > 推測` とする。
- DTO / Repository / Service / DI 分離などは、対象プロジェクトに既存パターンがある場合、またはユーザー要件で明示された場合のみ採用する。
- コスト防衛と仕様精度が衝突する場合は、推測で埋めずに未確定事項として止める。

## 作業順序
対象範囲が明確な場合は、以下の順番で進める。

1. 対象 root、ユーザー要件、成果物 root を確認する。
2. `project-reference-router` で reference 候補を選ぶ。見つからない場合は止まらず実コード調査へ進む。
3. `project-reference-intake` で reference 由来の仮説を整理する。reference がない場合はスキップする。
4. README、ソリューション、プロジェクトファイル、設定、テスト構成を必要最小限で確認する。
5. 対象機能が曖昧な場合は、`git status --short`、`git diff --name-only`、`.sln`、`csproj`、Route、Controller、API、Job、View、Test の順で候補を絞る。
6. 対象機能の入口と呼び出し関係を、代表的な実コードから調査する。
7. 現行仕様、入力、出力、副作用、エラー、ログ、設定、テストを根拠付きで整理する。
8. Reference 仮説と実コードの差分を整理する。
9. 要件と現行仕様の差分を整理する。
10. 実装仕様書を作る。
11. 実装計画書を作る。
12. 実装指示書を作る。
13. `progress/006-次回着手メモ.md` に、実装 skill が最初に読むべきファイルと最初の 1 手を明記する。
14. `progress/007-agent-skill-improvement-log.md` に、内部思考ではなく改善材料を残す。

対象が広い場合は、1〜6 の範囲でフェーズ 1 として停止し、追加調査候補、推定コスト、次に読むべき最小ファイルだけを提示する。

## 実装 skill への受け渡し契約
分析完了時は、`project-enhancement-implementation` がそのまま読めるように以下を固定する。

- `spec_root`: `.workstate/<Project>/<Feature>/spec/`
- `progress_root`: `.workstate/<Project>/<Feature>/progress/`
- `primary_inputs`: `spec/030-<Feature>実装指示書.md`, `spec/020-<Feature>実装計画書.md`
- `supporting_inputs`: `spec/010-<Feature>実装仕様書.md`, `spec/005-<Feature>現行仕様整理.md`, `spec/000-<Feature>前提メモ.md`
- `resume_entry`: `progress/006-次回着手メモ.md`
- `decision_log`: `progress/003-意思決定ログ.md`
- `verification_log`: `progress/005-検証メモ.md`
- `improvement_log`: `progress/007-agent-skill-improvement-log.md`

`030-<Feature>実装指示書.md` には、実装時に追加判断しなくてよい粒度で以下を必ず含める。

- 対象機能名と改修目的
- 変更してよい範囲
- 変更してはいけない範囲
- 実装順序
- 追加 / 修正対象ファイル候補
- テスト追加または検証引き継ぎの観点
- 禁止事項
- 未確定事項が残る場合の停止条件
- ビルド・テスト実行を人間へ引き継ぐ場合の確認コマンド候補
- 実装時に最低限見るべき静的確認観点

## 探索制約
- 初回調査では全ファイルを無差別に読まない。
- 原則として `.git`、`node_modules`、`bin`、`obj`、`dist`、`packages`、生成物、大容量ログを除外する。
- 初回は入口候補と関連ファイル中心に読む。
- 複数候補が残る場合は、深掘り前に要確認事項として提示する。
- 代表的な 3〜5 ファイルのサンプリングで判断できる範囲を先に示す。
- 100 行を超える読み込みやコマンド出力を避ける。

## 出力ルール
最終回答または最終ドキュメントの冒頭には、必ず以下を簡潔に書く。

- 対象機能名
- 改修目的
- 現行仕様の要点
- 実装時に最も注意すべき点
- reference_status
- 実コード優先で採用した判断
- spec_root
- progress_root
- 実装 skill への主入力
- ビルド・テスト実行を人間へ引き継ぐ前提

サンプリング分析で停止する場合は、以下を簡潔に書く。

- 調査した範囲
- 読んだ代表ファイル
- 現時点の仮説
- 追加調査が必要な理由
- 次に読むべき最小ファイル
- 続行時に増える可能性があるコスト

## 禁止事項
- 現行コードを十分に読まずに実装仕様を決めること。
- 別ディレクトリへ新規プロジェクトを作ること。
- 対象外の技術刷新や横断リファクタを混ぜること。
- ユーザー要件にない既存挙動変更を仕様へ入れること。
- 推測を確認済み事実として書くこと。
- 移行専用の成果物名や旧ソース root を要求すること。
- Reference 由来の仮説を、実コード確認なしに確定仕様へ入れること。
- 全ファイル精査、自律的な Explore、バックグラウンドエージェント起動を行うこと。
- 長文ファイルやコマンド結果を、分割せずに大量にコンテキストへ流し込むこと。
- 指示なくビルド、テスト、アプリ起動、DB 接続、外部 API 呼び出しを行うこと。
- progress ファイルを調査途中で高頻度に更新すること。
- 対象範囲が曖昧なまま、分析、仕様作成、実装、テスト実行を一括で進めること。
- 実装まで一括で進めること。
