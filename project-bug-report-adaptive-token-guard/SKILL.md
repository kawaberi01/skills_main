# Project Bug Report Adaptive Token Guard

不具合調査の精度を落とさず、トークン消費を抑えるためのガードレール。

## 目的

- 全文読みと全探索を避ける
- 入口候補を絞ってから必要箇所だけ読む
- 調査量ではなく、根拠の質を優先する
- 読めなかった範囲は隠さず `未確認` として扱う

## 基本上限

- 初回に読むファイルは最大5件
- 追加確認で読むファイルは最大3件
- 1回に読む本文は原則100行以内
- 1ファイルから読む範囲は、検索ヒット周辺または対象メソッド周辺に限定する
- reference は初回最大2件、追加確認で最大1件
- 横断検索は最大3クエリまで

## 大容量ファイルの扱い

以下に該当する場合、全文読みを禁止する。

- 500行を超えるコードファイル
- 200KBを超えるテキストファイル
- ログ、CSV、JSON、SQL、生成コード、minify済みファイル

代わりに以下を使う。

- ファイル名とパスだけ確認する
- `rg -n` で該当語を検索する
- ヒット行の前後20〜40行だけ読む
- メソッド名やクラス名が分かる場合は、その周辺だけ読む
- ログはエラー行、時刻、相関ID、例外名の周辺だけ読む

## 検索方針

検索語の優先順位:

1. エラーメッセージの固有部分
2. 画面名、API名、ジョブ名、テスト名
3. class、method、namespace
4. DBテーブル名、カラム名、設定キー
5. チケット中の業務語

検索結果は件数を増やすより、入口に近いものを優先する。

## 追加確認の停止条件

以下のいずれかに達したら、追加読み込みを止めてレポート化する。

- 対象プロジェクトが未確定のまま候補が3件以上残る
- 入口候補が5件を超え、絞り込み根拠がない
- 追加確認対象が3件を超える
- 原因候補が複数残り、データや環境確認なしに進めない
- 大容量ファイルの全文読みが必要になりそう

## レポートへの書き方

読めなかった範囲は、次のように明記する。

- `未確認: 大容量ログの全文確認は行っていない`
- `未確認: 入口候補が複数残るため、追加情報が必要`
- `限定確認: 検索ヒット周辺のみ確認`

## 禁止事項

- 大容量ファイルの全文読み
- 検索結果の全件要約
- 関連が薄いファイルのついで読み
- レポートを埋めるためだけの横断検索
- 根拠が弱い状態での工数見積もり
