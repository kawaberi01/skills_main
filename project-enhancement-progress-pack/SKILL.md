# Project Enhancement Progress Pack

この skill は、対象プロジェクト自身の改修作業を複数セッションにまたがって継続するために使う。  
目的は、仕様、計画、実装状況、判断、検証結果を `.workstate` に整理し、次回再開できる状態を保つことである。

## ディレクトリ方針
- 既定の進捗 root: `.workstate/<Project>/<Feature>/progress/`
- 既定の仕様 root: `.workstate/<Project>/<Feature>/spec/`
- `<Project>` は現在のワークスペース名、またはユーザーが明示したプロジェクト名を使う。
- `.workstate` は作業メモであり、対象ソースコードと混同しない。

## spec 標準構成
- `000-<Feature>前提メモ.md`
- `005-<Feature>現行仕様整理.md`
- `010-<Feature>実装仕様書.md`
- `020-<Feature>実装計画書.md`
- `030-<Feature>実装指示書.md`

## progress 標準構成
- `001-詳細タスク.md`
- `002-進捗サマリ.md`
- `003-意思決定ログ.md`
- `004-保留事項一覧.md`
- `005-検証メモ.md`
- `006-次回着手メモ.md`
- `007-agent-skill-improvement-log.md`

## 再開時の読み順
1. `progress/006-次回着手メモ.md`
2. `progress/002-進捗サマリ.md`
3. `progress/003-意思決定ログ.md`
4. `progress/001-詳細タスク.md`
5. `progress/007-agent-skill-improvement-log.md`
6. `spec/030-<Feature>実装指示書.md`
7. `spec/020-<Feature>実装計画書.md`
8. `spec/010-<Feature>実装仕様書.md`
9. `spec/005-<Feature>現行仕様整理.md`
10. `spec/000-<Feature>前提メモ.md`

## 更新方式
`progress/` はファイルごとに更新方式を分ける。すべてを追記にしない。

| ファイル | 更新方式 | ルール |
| --- | --- | --- |
| `001-詳細タスク.md` | 更新型 | 既存タスクのステータス、依存、DoD を最新化する。必要な新規タスクだけ追記する。 |
| `002-進捗サマリ.md` | 更新型 | 現在地、完了済み、未完、次候補を最新状態に置き換える。履歴は持たせない。 |
| `003-意思決定ログ.md` | 追記型 | 判断が発生した単位で日時付きセクションを追記し、既存判断を削除しない。 |
| `004-保留事項一覧.md` | 更新型 | 保留項目の状態を最新化する。新規保留や別トラック項目だけ追記する。 |
| `005-検証メモ.md` | 追記型 | 検証実行ごとに日時付きセクションを追記し、既知失敗の解消は結果として追記する。 |
| `006-次回着手メモ.md` | 更新型 | 次回の最初の 1 手だけを最新化する。古い次回メモは残さない。 |
| `007-agent-skill-improvement-log.md` | 追記型 | セッションごとに日時付き評価を追記し、既存ログを削除しない。 |

`spec/` は基準文書であり、原則として日々のログを追記しない。仕様変更が必要な場合のみ内容を改訂し、改訂理由は `progress/003-意思決定ログ.md` に追記する。

## 更新ルール
- `spec/` は仕様や計画の基準を置く場所とし、日々の進捗ログを書き込まない。
- `progress/` はセッション中の状態、判断、検証結果、次回着手点を置く。
- `progress/` の更新時は、更新型 / 追記型の区別を守り、履歴ログを更新型ファイルへ溜め込まない。
- 判断や変更が発生したら `003-意思決定ログ.md` に日時付きで追記する。
- テスト結果や未実施理由は `005-検証メモ.md` に日時付きで追記する。
- セッション終了時は `002-進捗サマリ.md` と `006-次回着手メモ.md` を最新状態へ更新する。
- skill / agent の改善材料は、内部思考ではなく判断ログとして `007-agent-skill-improvement-log.md` に残す。
- `007-agent-skill-improvement-log.md` はセッションごとに追記し、エージェントが正しく動いたか、どの指示や skill を直すべきかを判断できる事実、根拠、影響、再発防止を残す。

## 実施ルール
- feature 名が不明な場合は、候補を探索してからユーザーへ確認する。
- 詳細な内部思考ではなく、次回再開者が追える判断要約を残す。
- 仕様ファイルと進捗ファイルの役割を混ぜない。
- `007-agent-skill-improvement-log.md` には内部思考を残さず、動作評価、逸脱、原因分類、改善アクション、次回確認観点だけを記録する。
