# Project Enhancement Implementation Spec

この skill は、現行仕様整理とユーザー要件を入力として、実装エージェントが迷わず着手できる仕様・計画・指示を作る。  
目的は、実装前に変更後仕様、変更範囲、対象外、テスト観点、禁止事項を固定することである。

## ゴール
- `000-<Feature>前提メモ.md` を作る。
- `010-<Feature>実装仕様書.md` を作る。
- `020-<Feature>実装計画書.md` を作る。
- `030-<Feature>実装指示書.md` を作る。
- 実装に進む前に、判断が必要な仕様差分を明確にする。

## 前提
- 対象 root は現在のワークスペース root を既定とする。
- 既存プロジェクトへ直接改修する前提で書く。
- 新規プロジェクト作成、技術刷新、横断最適化は、ユーザー要件でない限り対象外にする。
- 既存の設計、命名、配置、テスト方式を優先する。
- 未調査の仮定は実装仕様として確定しない。
- Reference は仮説であり、実コードで裏取り済みの事実を仕様化の正とする。
- DTO / Repository / Service / DI 分離などは、対象プロジェクトの既存構成にある場合、またはユーザー要件で明示された場合のみ採用する。

## 仕様化手順
1. ユーザー要件、`project-code-grounding-analysis` の結果、`005-現行仕様整理` を突合する。
2. Reference 由来の仮説と実コードで確認した事実を分け、実コードで裏取り済みの内容だけを仕様へ採用する。
3. 変更後の機能仕様、入力、出力、副作用、エラー処理を決める。
4. 変更対象と対象外を明確にする。
5. 既存構成に合わせた配置先、既存構成上の担当、追加・修正ファイル候補を決める。
6. テスト観点、自動テスト可否、手動確認観点を決める。
7. 実装順序、DoD、レビュー観点、禁止事項を具体化する。

## 実施ルール
- 仕様書にない改善提案は「別タスク候補」へ分離する。
- 実装者が配置・順序・テスト観点を追加判断しなくてよい粒度にする。
- ただし、コード断片の大量生成ではなく、責務と変更契約を優先する。
- 実装配置は「理想アーキテクチャ」ではなく「対象プロジェクトの既存パターン」を優先して決める。
