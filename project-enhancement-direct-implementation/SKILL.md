# Project Enhancement Direct Implementation

この skill は、対象ワークスペースへ直接実装するために使う。  
目的は、確定済みの実装契約に従い、既存構成・命名・テスト方式を崩さずに改修を完了させることである。

## 前提
- 対象 root は現在のワークスペース root とする。
- 別ディレクトリに検証用または作業用プロジェクトを作らない。
- 既存プロジェクトの構成、命名、依存方向、設定方式、テスト方式に合わせる。
- 仕様書にない最適化、横断リファクタ、技術刷新を混ぜない。
- 機密情報、接続文字列、固定宛先、環境依存値をコードへ埋め込まない。
- 既存構成にない DTO / Repository / Service / DI 分離などを標準作成しない。
- 仕様資料と実コードが矛盾する場合は、実装で勝手に解消せず、要確認事項として止める。
- コミット、プッシュ、デプロイは明示指示がある場合のみ行う。

## 実装手順
1. 実装契約と関連コードを読み、変更対象を再確認する。
2. `030` に実装順序がある場合はそれを優先する。
3. `030` に順序がない場合は、既存変更単位を確認して順序を決める。
4. 変更単位は、対象プロジェクトに実在する Controller / View / ViewModel / Form / Entity / static model / helper / Service / Repository / Query / Command / Config / Test / Other を使う。
5. レガシー構成では、既存の Controller、ViewModel、Entity、static model、helper、View、設定の流れを崩さずに変更する。
6. 現代的構成では、既存の Service、UseCase、Repository、Query、Command、DTO、Config などの責務に合わせて変更する。
7. 新規抽象化は、既存類似パターンがある場合、または `030 / 010` に理由が明記されている場合のみ行う。
8. 既存テストがある場合は同じテストプロジェクト・フレームワークへ追加する。
9. 既存テストがない場合は、最小の検証可能単位と手動確認観点を明記する。
10. 実装後に仕様書、実装指示、実コード差分、テスト結果を突合する。

## 自己レビュー観点
- 指示された変更範囲だけを変更しているか。
- 既存の命名、配置、責務分担に合っているか。
- 既存挙動を意図せず変えていないか。
- エラー処理、ログ、設定、入力検証が既存方針に合っているか。
- テストまたは検証観点が今回の変更に対応しているか。
- 不要な大規模リファクタやフォーマット変更が混入していないか。
- 既存構成にない DTO / Repository / Service / DI 分離を理由なく追加していないか。
- 仕様資料と実コードの矛盾を確認なしに解消していないか。

## 完了条件
- 実装契約の変更対象が反映されている。
- 対象外作業が混入していない。
- 必要なテストまたは検証が実施、または未実施理由が明記されている。
- 変更内容、理由、検証結果、残課題を説明できる。
- `007-agent-skill-improvement-log.md` へ渡せる改善材料が整理されている。
