# PR説明文テンプレート

以下のマークダウン形式でPR説明文を生成する。各セクションは差分分析の結果に基づいて記述する。

---

## テンプレート

```markdown
## 概要
{変更の目的と影響を1〜3文で簡潔に記述}

## 修正内容
### {変更カテゴリ1}（例: ○○Manager.cs の構造改善）

- {変更の要点1}
- {変更の要点2}
- {変更の要点3}

### {変更カテゴリ2}（例: テストコード の追加）

- {テストファイル名} を新規作成
- {テスト内容の説明1}
- {テスト内容の説明2}

### {変更カテゴリ3}（例: パッケージ参照の追加）

- {パッケージ名} (Version {バージョン}) を追加

## 動作検証
- {検証項目1}
- {検証項目2}
- {検証項目3}

## 関連するRedmineチケット
{Redmine URLがある場合}
{URL}
重要度: {重要度}
緊急度: {緊急度}
期日: {期日 or 未設定}

{Redmine URLがない場合}
なし

## その他
{補足事項があれば記載。なければ空欄}
```

---

## 記述ガイドライン

### 概要
- 「何を」「なぜ」変更したかを明確にする
- 技術的な詳細よりもビジネスインパクトを優先して記述
- 例: 「医薬品の商品詳細ページに表示されるクチコミについて、「商品詳細表示不可」ステータスのレビューが誤って表示されていた点を修正しました。」

### 修正内容
- ファイル単位またはカテゴリ単位でグループ化する
- 新規メソッド・クラスの追加は明示的に記載する
- リファクタリングの場合は「抽出」「パラメータ化」等の具体的な手法を記載する
- テストコードは独立したセクションとして記載する
- パッケージ追加はバージョン番号を含める

### 動作検証
- テストコードで検証した項目を具体的に列挙する
- テストコードがない場合はユーザーに確認を促す
- 「〜が正しく動作していることを確認」「〜となることを確認」の形式で記載する

### 関連するRedmineチケット
- ブランチ名やコミットメッセージからチケット番号を推測する（例: `feature/18535-xxx` → issues/18535）
- Redmine MCPで取得できた場合はチケット情報（重要度・緊急度・期日）を記載する
- 取得できない場合はユーザーに確認するか「なし」とする

### その他
- レビュアーへの注意事項や補足がある場合に記載
- 特になければ空欄のままにする
