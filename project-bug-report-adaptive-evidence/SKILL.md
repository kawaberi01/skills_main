# Project Bug Report Adaptive Evidence

原因候補の根拠評価を行うスキル。

## 目的

- 症状説明をコード事実で裏取りする
- 推測を候補のまま分離する
- 根本原因に近いものから順に整理する
- `project-bug-report-adaptive-token-guard` の上限を超えない範囲で根拠を確認する

## 確認対象

- 入口メソッド
- 主な分岐条件
- バリデーション
- 例外処理
- 設定値
- データアクセス
- 外部サービス呼び出し
- 関連テスト、失敗テスト、ログ

## 手順

1. routing で絞られた入口候補から、正常系と異常系の分岐を追う
2. 症状と一致する失敗条件を探す
3. 原因候補を列挙し、`有力` / `代替` / `否定済み` / `要確認` に分ける
4. DB、設定、環境差分、外部依存が原因になり得る場合は、コード修正要否も分けて整理する
5. 最有力候補を1件に絞れない場合は、複数候補のまま保持する

## 読み込み制約

- 追加で読むファイルは最大3件
- 1ファイルから読む範囲は対象メソッド周辺または検索ヒット周辺に限定する
- 500行超のファイル、ログ、CSV、JSON、SQL は全文読みしない
- 全文確認できない場合は、確定原因ではなく `要確認` として扱う

## 出力

1. 調査した実行経路
2. 有力候補
3. 代替候補
4. 否定済み候補
5. 要確認事項
6. コード修正が必要そうかどうか
7. 次に追加で確認すると効果が高い1〜3点

## 出力ルール

- `事実`
- `推測`
- `要確認`

を分ける。

- 実コード、再現結果、テスト、ログで裏取りできないものは確定原因にしない
- 修正方針の比較や工数化へ先走らない
- 読み込み上限に達した場合は、追加確認事項として止める
