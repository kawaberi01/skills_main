# Project Bug Report Adaptive Intake

不具合調査を始める前に、入力情報を無駄なく整理するためのスキル。

## 目的

- チケット文面をそのまま信じず、調査観点へ変換する
- 後続で必ず裏取りすべき点を明確にする
- 情報不足でも調査開始できる最小構造を作る

## 受け取る入力

- チケット番号
- 件名
- 本文
- 期待値
- 実際値
- 発生画面 / API / バッチ / 機能名
- 再現条件
- エラーメッセージ、ログ、スタックトレース
- 添付画像から読める補助情報
- ユーザーが指定した調査対象外や制約

## 出力

以下だけを簡潔に整理する。

1. チケット要約
2. 期待値 / 実際値
3. 発生条件 / 再現手順
4. 調査の入口になりそうな固有語
5. 不足情報
6. 初回調査で確認すべき観点
7. 調査対象外と制約

## ルール

- 不足情報があっても即停止しない
- ただし、曖昧な点は `要確認` と明示する
- 原因や改修方針はまだ書かない
- レポート章立てをここで固定しない
- 出力は後続 skill がそのまま引用できる短い箇条書きにする

## 推奨出力形式

- `事実`
- `要確認`
- `初回調査観点`

各項目を3〜6行程度でまとめる。
