# ブランチ差分コードレビュー

**重要**: コード/リポジトリの変更は一切行わない。レビューのみ実施する。

## 前提条件
- レビュー対象ブランチがチェックアウト済みであること
- git が使用可能であること

## レビューワークフロー

### Step 1: 差分の取得

```bash
# 現在のブランチを確認
git branch --show-current

# masterとの分岐点を特定
git merge-base HEAD origin/master

# 変更ファイル一覧を取得
git diff <分岐点コミットID>..HEAD --name-status

# 変更の統計（行数の感覚把握）
git diff <分岐点コミットID>..HEAD --stat
```

ファイル数・規模を確認してから、**カテゴリ単位でまとめて差分を取得**する（1コマンドで複数ファイルを渡す）。

```bash
# 複数ファイルまとめて取得（例）
git diff <分岐点コミットID>..HEAD -- ファイル1 ファイル2 ファイル3
```

### Step 2: 差分の分析

差分のボリュームに応じて読み方を調整する：

| 規模 | 対応 |
|------|------|
| 5ファイル以下 | 全差分を一括取得して読む |
| 6〜20ファイル | カテゴリ（モデル層・ViewModel層など）に分けて取得 |
| 21ファイル以上 | 影響の大きいファイルを優先、残りは変更パターンを確認 |

差分が大きい場合は、呼び出し元の文脈を確認するため変更箇所の周辺コードも `read_file` で確認する。

### Step 3: レビュー実施

[references/checklist.md](references/checklist.md) のチェック項目に沿ってレビューする：

- **A. アーキテクチャ・設計** - 既存パターン準拠、責任分離
- **B. コード品質** - 命名規則（C#/.NET）、可読性、重複、不統一
- **C. 機能性・ロジック** - エッジケース、エラーハンドリング、論理バグ
- **D. パフォーマンス** - N+1問題、ループ内DB/キャッシュアクセス、SQL上限
- **E. セキュリティ** - SQLi、XSS、機密情報
- **F. テスト** - テストコードの有無・網羅性
- **G. 既存コードとの整合性** - 変更パターンの統一性、無関係差分の混入

#### ビジネスロジックに関する限界を認識する

以下はソースのみでは判断困難なため、「確認が必要」として明示する：

- ビジネスルール上の正しさ（仕様依存）
- マジックナンバーの意味（例: カテゴリID `13, 14` が何を表すか）
- キャッシュTTLの業務的な妥当性
- 実際のデータ量・実行頻度を要する性能見積もり

### Step 4: レビュー結果の出力

[references/checklist.md](references/checklist.md) の出力フォーマットに従い結果をまとめる。

ヘッダーには必ず以下を含める：
- ブランチ名
- merge-base コミットID
- 変更ファイル数・コミット数

最終判定: ✅承認 / ❌要修正 / ⚠️条件付き承認

## 注意事項

- **読み取り専用**: コードの修正は行わない（ユーザーから明示的な依頼があれば別）
- **網羅性**: 全変更ファイルを確認する（大規模な場合は優先度をつけて説明する）
- **建設的**: 問題の指摘だけでなく具体的な改善案を添える
- **客観性**: ベストプラクティスに基づいて評価し、主観的な好みで批判しない
- **不明点は明示**: ドメイン知識が必要な判断は「ソースのみでは確認困難」と明記する
