## 必須チェック項目

### A. アーキテクチャ・設計
- [ ] 既存のアーキテクチャパターンに準拠しているか
- [ ] SOLID原則に従っているか
- [ ] 適切な責任分離がされているか
- [ ] 依存関係が適切に管理されているか

### B. コード品質
- [ ] 命名規則が統一されているか（C#/.NET規約準拠）
- [ ] コードの可読性が十分か
- [ ] 重複コードがないか
- [ ] コメントスタイルが統一されているか（日付有無、形式など）
- [ ] 不要・無関係な差分が混入していないか（空白変更、無関係リファクタ）
- [ ] マジックナンバーの使用がないか

### C. 機能性・ロジック
- [ ] 要件を正しく実装しているか
- [ ] エッジケースが考慮されているか（null、空リスト、0件など）
- [ ] キャッシュのヒット/ミス判定ロジックが正しいか
- [ ] Negative cache（「該当なし」のキャッシュ）が考慮されているか
- [ ] エラーハンドリングが適切か
- [ ] 変更前にあった絞り込み・JOIN条件が意図的に削除されていることを確認したか

### D. パフォーマンス
- [ ] ループ内での DB / キャッシュアクセスがないか（N+1）
- [ ] 同一データの重複取得がないか（同ファイル内で複数回同じ呼び出し）
- [ ] SQL `IN` 句が大量パラメータになる可能性はないか（SQL Server上限 ~2100）
- [ ] 大量IDを扱う場合のバッチ分割が考慮されているか
- [ ] メモリリークの可能性はないか

### E. セキュリティ
- [ ] SQLインジェクションの脆弱性はないか
- [ ] XSS攻撃への対策がされているか
- [ ] 認証・認可の実装が適切か
- [ ] 機密情報のハードコードがないか

### F. テスト
- [ ] 単体テストが追加・更新されているか
- [ ] エッジケース（空リスト、重複ID、null）のテストがあるか
- [ ] キャッシュ系の変更に対してキャッシュ動作のテストがあるか

### G. 既存コードとの整合性
- [ ] 同種の変更が全対象ファイルで均一に適用されているか
- [ ] 既存のヘルパー・ユーティリティを適切に使用しているか
- [ ] `[Obsolete]` など非推奨属性が正しく付与されているか
- [ ] まだ残存している旧メソッドの呼び出し箇所がないか
