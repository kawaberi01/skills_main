# Project Code Grounding Analysis

この skill は、仕様策定で最も重要な実コード裏取りを担当する。  
原則は **Reference-first, Code-authoritative** とする。

## ゴール
- 対象機能の現行挙動を実コードから確認する。
- Reference 由来の仮説を裏取りし、差分を記録する。
- 入口、呼び出し経路、入力、出力、副作用、設定、テストを根拠付きで整理する。
- 最終判断の優先順位を `実コード > reference > 推測` に固定する。

## 前提
- Reference は調査の起点であり、仕様判断の正本ではない。
- 実コードで裏取りできない内容を確定仕様にしない。
- Reference と実コードが矛盾する場合は実コードを優先する。
- 重要な仕様判断には、根拠ファイルと可能なら行番号を添える。

## 探索制約
初回調査では全ファイルを無差別に読まない。

原則除外:
- `.git`
- `node_modules`
- `bin`
- `obj`
- `dist`
- `packages`
- 生成物、大容量ログ、ビルド成果物

優先探索:
1. `git status --short` と `git diff --name-only`
2. `.sln`、`*.csproj`、README、主要設定
3. Route / Controller / API / Job / View / Test
4. 関連 ViewModel / Form / Entity / static model / helper / DTO（既存にある場合のみ）
5. 設定キー、外部連携、DB 更新箇所

対象機能が曖昧で候補が複数残る場合は、深掘り前にユーザー確認が必要な候補として出す。

## 段階的な読み取り拡張
- 初回は入口候補を 5 件程度までに絞り、対象機能の当たりを付ける。
- 5 件は固定上限ではなく、無差別探索を避けるための初期バッチ目安とする。
- 次のいずれかに該当する場合は、理由を記録したうえで追加バッチを読む。
  - 呼び出し経路が途中で途切れており、入力 / 出力 / 副作用を説明できない。
  - Controller / API / Job など入口が複数あり、同一機能として連動している可能性が高い。
  - View / ViewModel / Entity / Service / Repository / helper / SQL / Config など、入口から直接参照される未確認ファイルが残っている。
  - Reference 由来の仮説と実コードの差分を判断する根拠が不足している。
  - テストまたは手動確認観点を決めるために、既存テストや設定の確認が必要である。
- 追加バッチでも関係が薄いファイルは深追いしない。
- 候補が広がり続ける場合は、これ以上読む候補と理由を要確認事項として提示し、深掘り前にユーザー確認する。

## 読み取り手順
1. `project-reference-router` と `project-reference-intake` の結果を確認する。reference がない場合は空でよい。
2. 要件語、プロジェクト構成、入口候補から対象機能の候補を絞る。
3. 入口ファイルを初期バッチとして 5 件程度まで優先して読む。
4. 入口から呼ばれる範囲に限定して、処理経路を追う。
5. 根拠不足がある場合は、段階的な読み取り拡張ルールに従って追加バッチを読む。
6. 入力、出力、副作用、設定、テストを整理する。
7. Reference 仮説と実コードの一致 / 差分を記録する。
8. 実コードで裏取り済みの判断だけを仕様化へ渡す。

## 実施ルール
- DTO / Repository / Service などの現代的構成は、既存にある場合のみ前提にする。
- レガシー構成では、Controller、ViewModel、Form、Entity、static model、helper、View など実在する構成を優先して表現する。
- 読み足りない箇所と読み過ぎた箇所は、改善ログへ渡せるように記録する。
